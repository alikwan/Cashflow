## 1. المنهجية — كيف يُعاد التحقّق

الفكرة: نشتقّ نفس المؤشّر **بطريقتين مستقلّتين** ونقارنهما:

| الطريق | المصدر | الأداة |
|--------|--------|--------|
| **(أ) الخام** | استعلام مباشر على `AlBaytAlSaeid` (SQL Server) | اتصال القراءة-فقط في الخلفية + استعلامات `app/etl/extract.py` |
| **(ب) النظام** | قراءة جداول PostgreSQL التحليلية أو ردّ الـ API | `monthly_cashflow` · `balances_snapshot` · `installments_summary` · `installments_aging` · `per_supplier_monthly` |

إن طابق الطريقان (ضمن أخطاء التقريب) فالنظام سليم.

### 1.1 البيئة

```bash
# من جذر المستودع:
cd "/Users/ak/Documents/sqlserver-docker/Monthly cash flow/cashflow-web/backend"
# تأكّد أنّ حاوية SQL Server تعمل (ETL والتحقّق يحتاجانها):
docker ps | grep mssql-server
# مفسّر بايثون الخلفية (فيه pymssql + pandas + التبعيات):
.venv/bin/python
```

### 1.2 الطريق (أ) — الأرقام الخام من `AlBaytAlSaeid`

داخل مفسّر الخلفية، الاتصال جاهز عبر دالّة الـ ETL نفسها (تستخدم مستخدم القراءة-فقط
من `.env`)، واستعلامات الاستخراج المعتمدة هي ذاتها التي يستهلكها النظام يومياً —
فلا انحراف بين «ما نتحقّق منه» و«ما يُحمَّل»:

```python
from app.etl.extract import (
    connect_mssql,            # اتصال pymssql للقراءة-فقط (من إعدادات .env)
    baghdad_today,            # «اليوم» بتوقيت بغداد — يقطع السندات المستقبلية والأعمار
    fetch_bonds,              # السندات الخام في نافذة [start, end)
    fetch_balances,           # Σ tAccounts.Balance حسب AccountTypeId
    fetch_installments_summary,   # صيغة DC-System: اسمي/نقد/خصم/قائم
    fetch_installments_aging,     # القائم لكل شريحة عمرية كما في asof
    fetch_per_supplier_monthly,   # مدفوعات/مقبوضات كل مورد شهرياً
)

conn = connect_mssql()
asof = baghdad_today()

# 1) مجاميع Bonds الخام حسب OperationsType (آخر 12 شهر، من الصناديق السبعة):
bonds = fetch_bonds(conn, start="2025-06-01", end_exclusive="2026-06-01")
#   مقبوضات  = Σ Amount1 حيث OperationsType==0
#   مصروفات شامل = Σ Amount1 حيث OperationsType ∈ (1,5,7)
#   (التفصيل بالفئة = OperationsType + نوع الحساب المقابل to_type — انظر classify.py)

# 2) رصيد الصناديق (صناديق النقد فقط):
bal = fetch_balances(conn, account_type_ids=[1811, 1812])   # Σ Balance

# 3) ملخّص الأقساط (صيغة DC-System — لا «مقلوب»):
summ = fetch_installments_summary(conn)        # face_total / cash_paid / discount / remaining

# 4) أعمار الديون كما في «اليوم»:
aging = fetch_installments_aging(conn, asof)   # القائم لكل شريحة
```

> ⚠️ **قواعد المحاسبة الإلزامية** (مطبّقة داخل استعلامات `extract.py`): `Amount1`
> بالدينار دائماً (لا يُضرب بـ `Rate1`)؛ دائماً `Deleted=0 AND ISNULL(IsEdit,0)=0`؛
> استبعاد السندات المستقبلية؛ التصنيف عبر `OperationsType` + الحساب المقابل (يلتقط
> الصيرفة Type 7). صيغة الأقساط: `PremiumPays.Amount` = **الرصيد المتبقّي للقسط لا
> المدفوع** (انظر `CLAUDE.md` §6).

### 1.3 الطريق (ب) — أرقام النظام

```bash
# جداول PostgreSQL التحليلية (تملؤها آخر دورة ETL):
docker exec cashflow-postgres psql -U cashflow -d cashflow -c \
  "SELECT SUM(cash_in_m) AS in_12, SUM(out_total_comprehensive_m) AS out_12
   FROM monthly_cashflow
   WHERE year_month >= '2025-06' AND year_month <= '2026-05';"
docker exec cashflow-postgres psql -U cashflow -d cashflow -c \
  "SELECT SUM(balance_m) FROM balances_snapshot;"          -- رصيد الصناديق
docker exec cashflow-postgres psql -U cashflow -d cashflow -c \
  "SELECT * FROM installments_summary;"                    -- اسمي/نقد/خصم/قائم
```

أو عبر الـ API بعد تسجيل الدخول (كوكي الجلسة) — نقاط النهاية للوحة، الأقساط،
الموردين، الأعمار. النتيجة يجب أن تطابق الطريق (أ).

> ملاحظة: نافذة «آخر 12 شهر» في الأمثلة (2025-06 → 2026-05) ثابتة هنا للتوثيق؛
> النظام يحسبها ديناميكياً من «اليوم» بتوقيت بغداد، فالحدود تتقدّم مع الزمن.

## 2. الأرقام المؤكَّدة (الخطّ المرجعي — Baseline)

> سُجّلت عند التحقّق الحيّ (2026-06). **كلّها طابقت المصدر الخام تماماً.**
> اعتمدها كخطّ مرجعي للمقارنة المستقبلية (مع مراعاة أنّ الأرقام المعتمدة على
> التاريخ تتغيّر — انظر §5). كل القيم بملايين الدنانير (M) ما لم يُذكر غير ذلك.

### 2.1 رصيد الصناديق — **225.15M** ✓

= Σ `tAccounts.Balance` للحسابات `accounts.AccountTypeId IN (1811, 1812)`:

| الصندوق | AccountId | الرصيد (M) |
|--------|-----------|-----------:|
| صندوق المعتصم (الرئيسي) | 181 | 123.32 |
| صندوق الضلوعية | 4935 | 69.93 |
| صندوق عمر حردان | 6684 | 31.90 |
| الباقي (180, 6314, 6662, 6672 …) | — | 0.00 |
| **الإجمالي** | | **225.15** |

> هذا **رقم فعلي من المحاسبة** (`Balance` المثبّت في `tAccounts`)، **ليس back-solve**.
> الـ back-solve يُستخدم فقط لاشتقاق **الرصيد الافتتاحي التاريخي** لسلسلة التدفّق،
> لا للرصيد الحالي. الرصيد الحالي في النظام (`balances_snapshot` / `current_cash_m`)

## 3. التحقّق اليدوي للمالك (المتّفق عليه — `CLAUDE.md` §4، البند 5)

النظام **لا يفرض** بوابة مطابقة آلية مانعة؛ بل يدعم تحقّق المالك اليدوي:

1. **رصيد الصناديق:** قارِن `current_cash_m` المعروض في النظام (**225M**) بما تعرفه
   فعلاً عن صناديقك (المعتصم + الضلوعية + عمر حردان). أي فرق كبير = راجع آخر دورة ETL
   ووضع `tAccounts.Balance`.
2. **الأقساط القائمة:** قارِن **1315M** القائمة (و~79% تحصيل) بإحساسك عن محفظة
   الديون. الرقم سليم محاسبياً (صيغة DC-System) لا «4.67 مليار».
3. **بقايا المطابقة:** `reconciliation_residual_m = 0` حالياً (الرصيد الافتتاحي
   مُشتقّ بالـ back-solve فيصفّر البقايا بحكم البناء). **العتبة المقبولة للفرق
   تُضبط لاحقاً** عندما تتراكم **لقطات أرصدة تاريخية** فعلية تسمح بقياس انحراف حقيقي.
   لا تُفسَّر القيمة 0 الحالية كـ«مطابقة مثالية مُثبَتة» — بل كغياب مرجع تاريخي بعد.

## 4. (ملخّص) ماذا تفعل إن لم يطابق رقم

1. تأكّد أنّ آخر ETL نجح: `GET /api/etl/status` أو جدول `etl_runs` (انظر RUNBOOK §3).
2. أعِد اشتقاق الرقم بالطريق (أ) أعلاه وقارنه بالطريق (ب).
3. إن اختلف الطريقان جوهرياً (وليس مجرّد تقدّم «اليوم» — §5) فهي حالة تستحقّ
   الفحص: راجع استعلام `extract.py` المعنيّ ومنطق `domain/` المقابل.

## 5. ملاحظة مهمّة — الأرقام المعتمدة على التاريخ تتغيّر (سلوك صحيح)

المؤشّرات المرتبطة بـ«اليوم» **يُفترَض أن تتغيّر** مع تقدّم الزمن، وهذا **سليم**:

- **أعمار الديون:** قسط لم يُسوَّ ينتقل من شريحة لأخرى كلّما تقادم (1–30 → 31–60 …).
- **نافذة آخر 12 شهر:** تنزلق شهرياً، فتدخل أشهر وتخرج أخرى.

لذلك أرقام §2 خطّ مرجعي **للحظة التحقّق** (2026-06). عند إعادة التحقّق لاحقاً، توقّع
انزياحاً طبيعياً في هذه البنود؛ ما يجب أن يبقى **متطابقاً دائماً** هو تساوي الطريق (أ)
مع الطريق (ب) في نفس اللحظة.
