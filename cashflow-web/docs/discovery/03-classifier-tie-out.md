# 03 — مطابقة التصنيف الأولية (مهمة تحقّق إلزامية A4)

**التاريخ:** 2026-06-02 | **شهر العيّنة:** 2026-04

## حركة الصناديق (1811/1812) حسب OperationsType

| OpType | داخل الصناديق (مليون) | خارج الصناديق (مليون) |
|--------|------------------------|------------------------|
| 0 (مقبوضات) | 97.3 | 0 |
| 1 (مدفوعات) | 0 | 80.9 |
| 5 (مصاريف) | 0 | 1.1 |

- صافي حركة الصناديق للشهر ≈ 97.3 − 80.9 − 1.1 = **+15.3 مليون**.
- لا حركات Type 7 (صيرفة) أو Type 3 (تحويل داخلي) لمست الصناديق في هذا الشهر (الصيرفة قد تمرّ عبر حسابات وسيطة لا الصناديق مباشرة).

## الخلاصة للمطابقة (§4.3)
- المطابقة per-OpType **ممكنة وسريعة**.
- لكن **الفرق الدقيق (residual)** بين الصافي المُصنَّف وتغيّر رصيد الصناديق الفعلي يحتاج **تأريخ أرصدة الصناديق** (وهي لقطة حالية فقط في `tAccounts`، لا تاريخ). لذلك:
  - يُحسب الرصيد الافتتاحي بالـ back-solve (الرصيد الحالي − Σ الصافي) كما في §4.3.
  - تُعرض `reconciliation_residual_m` كأداة **تدعم تحقّق المالك اليدوي**، لا كرقم محاسبي مُحكم.
- **عتبة تنبيه مقترحة** لـ `reconciliation_gap`: ابدأ بـ ±25 مليون/شهر (تُضبط بعد أول تشغيل فعلي كامل).

### الأثر
يؤكّد قرار §4.3: التصنيف بـ OperationsType سليم؛ والمطابقة معينة بصرية للمالك لا بوابة صارمة.
