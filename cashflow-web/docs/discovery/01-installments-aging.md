# 01 — حقول استحقاق الأقساط (مهمة تحقّق إلزامية A2)

**التاريخ:** 2026-06-02

## النتيجة: ⚠️ لا توجد حقول "تاريخ استحقاق" لكل قسط

**أعمدة `Premiums`:** Id, AccountId, BillId, CurrencyId, Date, Deleted, FirstPay, **InitDate**, MachineDate, Note, **PremiumNo**, PremiumPayAmount, **PremiumType**, RowVersion, SponsorId, TotalAmount, UserId, QCard, DistributorId, Note2.

**أعمدة `PremiumPays`:** Id, AccountId, Amount, CurrencyId, Date, Deleted, PremiumId, PremiumState, RowVersion, UserId, index, QCardId, EnjazBillID, EnjazgroupID, Account2Id, Account2Name, **DatePay**, Discount.

- التواريخ المتاحة كلها **إنشاء/دفع** (`Date`, `InitDate`, `MachineDate`, `DatePay`) — **لا عمود due-date** لكل قسط.
- `PremiumType` توزيعه: النوع `1` = 19,715 (≈98.6%)، وأنواع 2–5 قليلة جداً. لا يوجد جدول جدولة استحقاقات مستقل.

## ✅ الحالة: محلولة — الأعمار قابلة للحساب (البيانات موجودة)

**التصحيح:** الاستنتاج الأول كان خاطئاً. تواريخ الاستحقاق **مشتقّة** (لا عمود صريح) — وهذا هو النموذج المعياري لأنظمة التقسيط. المالك محقّ: البيانات موجودة.

### النموذج المؤكَّد (من فحص فعلي 2026-06-02)
- `Premiums` = **صف واحد لكل فاتورة = خطة قسط** (20,005 فاتورة بصف واحد).
- **`PremiumNo` × `PremiumPayAmount` = `TotalAmount`** (مؤكَّد بالعيّنات: خطة بـ N قسطاً شهرياً متساوياً).
- **`PremiumType=1`** (98.6%) = **شهري**؛ **`InitDate` = استحقاق أوّل قسط** (العقد بـ`Date`، والاستحقاق الأول بعده بشهر).
- جدولة القسط k (k=1..N): `DATEADD(MONTH, k-1, InitDate)` بقيمة `PremiumPayAmount`.
- `PremiumPays` = الدفعات الفعلية (FIFO تغطّي أقدم الأقساط).

### الإجماليات الفعلية (2026-06-02)
ملتزم **7.12 مليار** · مدفوع **1.26 مليار** · متبقٍ **≈5.86 مليار** (الـ4.67 في `CLAUDE.md` كانت بفلتر/لقطة سابقة).

### حساب الأعمار (أُثبت بـ SQL — الشرائح تجمع للمتبقّي)
SQL مرجعي: لكل خطة، عدد الأقساط المغطّاة = `FLOOR(paid/PayAmt)`، فأقدم قسط غير مدفوع يستحق `DATEADD(MONTH, covered, InitDate)`، والعمر = أيام حتى اليوم → الشريحة.

### ⚠️ قرار تعريف العمر (يحتاج المالك)
الأعمار بالجدولة **الصارمة** تُظهر **~98% (≈5.7 مليار) في «+120 يوم»** — لأن العقود طويلة والزبائن يدفعون متأخّرين عن الجدول (طبيعي هنا)، فلا تعطي إنذاراً مفيداً. التوزيع حسب **آخر دفعة** أكثر دلالة:

| آخر دفعة | متبقٍ (مليار) | الدلالة |
|---------|----------------|---------|
| 2026 | ~0.62 | نشط يدفع |
| 2024–2025 | ~2.58 | يدفع ببطء |
| 2020–2022 | ~1.4 | راكد/متعثّر محتمل |

## ✅ القرار المعتمد: مزيج (جدولة + عزل الراكد)

اختار المالك **(ج) المزيج**: العقود بلا دفعة منذ **> 12 شهراً = «راكدة»** تُعرض كفئة منفصلة؛ والنشطة تُعمَّر بالجدولة الصارمة (أقدم قسط غير مدفوع).

**النتيجة الفعلية** (today=2026-06-02، عتبة الركود = آخر نشاط < 2025-06-02):

| الفئة (`bucket_key`) | خطط | متبقٍ (مليار) |
|------|-----|----------------|
| `current` (لم يستحق) | 652 | 0.063 |
| `b0_30` | 107 | 0.017 |
| `b31_60` | 111 | 0.020 |
| `b61_90` | 103 | 0.019 |
| `b91_120` | 103 | 0.021 |
| `b120` (نشط متأخّر) | 3,051 | 1.103 |
| **`stale`** (>12 شهر بلا دفعة) | **15,154** | **4.620** |

**رؤية مهمة:** ≈**4.62 مليار راكدة** من أصل 5.86 مليار "مستحقة" — مال متوقّف فعلياً؛ النشط القابل للتحصيل ≈1.24 مليار.

### مواصفة `installments_aging` النهائية
- `bucket_key ∈ {current, b0_30, b31_60, b61_90, b91_120, b120, stale}`.
- `stale` = `COALESCE(آخر دفعة, InitDate) < today − 12 شهراً` (مع `remaining>0`).
- النشطة: العمر = أيام منذ استحقاق أقدم قسط غير مدفوع = `DATEADD(MONTH, FLOOR(paid/PremiumPayAmount), InitDate)`.
- يفرض ETL Σ(amount عبر كل الفئات) = `installments_summary.remaining_m`.


### الأثر على المواصفة/الخطط
- **يُسقط جدول `installments_aging`** من النسخة الأولى (المواصفة §3.1) — أو يُترك فارغاً.
- صفحة `Installments` (الخطة 3 / تصميم `data.js` `AGING`): تُخفى لوحة الأعمار، أو تُستبدل بملاحظة "غير متاحة (لا حقول استحقاق في المصدر)".
- **مستقبلاً:** يمكن اشتقاق استحقاقات تقريبية من `InitDate` + `PremiumNo` + دورية `PremiumType` (إن تأكّدت دلالتها) — خارج نطاق v1.
